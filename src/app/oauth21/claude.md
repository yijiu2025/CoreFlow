## 用户认证逻辑设计指南

# 核心架构概览

最新的OAuth认证架构不再仅仅是“授权”，而是一个集 身份认证（Authentication）、授权（Authorization） 和会话管理于一体的统一安全体系。其核心组件与分层如下：

1.客户端层 (Client Layer)：请求访问资源的应用程序（如Web前端、移动端App、CLI工具）。2.授权服务器层 (Authorization Server Layer)：系统的安全中枢，负责身份验证、颁发和管理令牌。它通常集成了 OpenID Provider (OP) 功能以支持OIDC。3.资源服务器层 (Resource Server Layer)：托管受保护资源的服务器（如API服务），负责验证访问令牌并执行授权决策。4.身份与权限提供者层 (Identity & Permission Provider Layer)：授权服务器的后端支撑，负责管理用户身份、角色、权限等信息（如IAM系统、LDAP、数据库）。

当前为授权服务器层负责安全中枢，负责身份验证、颁发和管理令牌

对外提供api(登录->返回)

# 安全架构核心组件

基于Spring Security 2026等现代框架
，一个健壮的OAuth架构包含以下关键安全部件：

1.授权服务器：可采用 Spring Authorization Server 等成熟开源方案，它完整支持OAuth 2.1和OIDC
。2.资源服务器：配置为OAuth2资源服务器，能够验证JWT格式的访问令牌
。3.权限管理：OAuth 2.0本身不管理用户权限的来源
。权限信息（如roles, permissions）通常由授权服务器在颁发令牌时，根据后端IAM系统的数据嵌入到JWT的Claims中
。资源服务器根据这些Claims进行细粒度的访问控制。4.客户端管理：在授权服务器中注册客户端信息，包括client_id、client_secret、支持的授权类型、回调地址等
。

# 适配不同场景的授权模式

除了核心的授权码流程，架构还需支持其他标准模式以满足特定需求：

客户端凭证模式 (Client Credentials)：适用于服务间通信（M2M）。客户端直接使用自己的凭证（client_id和client_secret）向授权服务器请求访问令牌，不涉及用户交互
。
设备码模式 (Device Code)：适用于输入受限设备。设备显示一个代码和验证URL，用户在另一设备（如手机）上访问URL并输入代码以完成授权
。
JWT持有者授权 (JWT Bearer Grant)：适用于服务账户认证。客户端使用一个预先签名的JWT作为断言来获取访问令牌，常用于CI/CD流水线或后台任务
。

# 一个最新的、生产就绪的OAuth认证结构应具备以下特征：

协议基础：基于 OAuth 2.1 和 OIDC 1.0。
安全核心：对公共客户端强制实施 PKCE，采用 JWT 作为访问令牌格式。
架构组件：清晰的授权服务器、资源服务器、客户端三层分离。
令牌策略：短期访问令牌搭配长期刷新令牌。
场景覆盖：支持授权码+PKCE、客户端凭证、设备码等主要授权模式。
权限集成：通过JWT Claims将后端IAM系统的权限信息无缝传递到资源服务器
。
