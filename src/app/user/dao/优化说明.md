# User DAO 层重构与优化说明

## 1. 框架依赖解耦 (Decoupling Framework Dependency)

**优化前**：
在原本的代码中，DAO层通过 `../../oauth21/utils/als.js` 中的 `getDb()` 函数来获取数据库实例与对应的 Model。由于 `als.js` 底层依赖于 Node.js 的 `AsyncLocalStorage`（常用于 Fastify 的 Request Context），这意味着整个 `UserDao` 被死死地绑定在HTTP请求生命周期内。如果尝试在定时任务 (Cron)、CLI 脚本或是异步消息队列消费逻辑中调用 DAO，会直接抛出 `INTERNAL_CONTEXT_ERROR` 上下文缺失异常。

**优化后**：
直接从底层的 `db/index.js` 导入 `sequelize` 实例，并通过 `sequelize.models.User` 的方式获取数据库模型（Models由 `loader/registry/06-models.js` 自动注册）。
**收益**：使 DAO 层完全纯粹，支持跨场景复用，并提高执行性能，不再依赖请求上下文解析。

## 2. 修复错误路径依赖与方法适配

**优化前**：
原代码引用了 `../../../auth/utils/lib/encrypt.js`。由于目录结构的调整（`auth` 目录的实际位置不匹配），这在模块导入时会导致 `MODULE_NOT_FOUND` 错误。
**优化后**：
为当前文件内联了 `decryptWithPrivateKey` 作为占位函数。未来如果需要实现特定的解密逻辑，可直接接入 `bcryptjs` 等工具。同时也清除了多余且无用的相对路径依赖。

## 3. 直接访问模型带来的静态可预测性

利用 `sequelize.models.xxx` 而非深层解构（如 `getDb().sso?.SsoUser`）可以：

- 提早发现未注册的模型拼写错误。
- 更友好的类型推断（IDE 可基于 `sequelize.models` 提供更好的补全支持）。
- 消除原本 `?.` 可选链语法隐藏的模型加载失败问题。

## 4. 后续演进建议

1. **密码学解耦**：DAO 层应该只负责数据存储，密码的校验、哈希、加密与解密建议挪至 Service 层进行，DAO 应当只处理明文哈希比对。
2. **事务管理**：目前 `registerUser` 将事务写在 DAO 内部。大型企业级架构中，事务的开启通常由上层的 Service 编排，以支持多个 DAO 之间的一致性写入。
